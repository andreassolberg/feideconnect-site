Mobile applications

OAuth 2.0 is designed to work well with mobile applications, both native and web applications.

Typically the application will initate login, opening a web view where the user enters credendials, before Connect is issuing the access token and sending it back to the mobile app.

The app will use the token for looking up information about who the user is, and may be communicate with its backend through the Connect API Gatekeeper.

Authentication

There are more than one approach to deal with how a native app opens the authorization web view for the user. We strongly reccomend to use the native browser for this.

System-wide browser and custom URL schemes

To initate the OAuth 2.0 flow, open the native browser app to open the OAuth authorization URL.

Register a custom URL Scheme for the native app, such as myconnectapp.

In the Connect dashboard, regsiter a redirect URI such as myconnectapp://.

Then register a listener that is triggered when the app is opened with this scheme, and then deal with receiving the OAuth token.

In-app browser

Alternatively, the mobile OS typically offers a web component that might be used within the app.

We would not reccomend this option. Mostly because of the security flaw of the user not beeing able to verify the ssl connection to Feide, and having to trust that the app is not intercepting and stealing the password.

Protecting access to backend APIs

The Connect API Gatekeeper is very well suited to handle access control to the API backend of your client.
